# 바이브코딩 실습 매뉴얼 — KOStockCrewAI

> 이 문서는 **비개발자 교육생**이 "바이브코딩(Vibe Coding)"으로 한국 주식 분석 앱(KOStockCrewAI)을
> **처음부터 끝까지 스스로 따라 만들 수 있도록** 정리한 실습 가이드입니다.
> 실제로 진행한 과정을 단계별 프롬프트와 함께 재현 형식으로 담았습니다.

---

## 0. 학습 목표

이 실습을 마치면 교육생은 다음을 할 수 있습니다.

1. **바이브코딩의 원리**(프롬프트=명세서, 실행→오류→수정의 반복 루프)를 이해한다.
2. **좋은 프롬프트의 필요충분조건**을 설명하고, 자신의 프롬프트에 적용한다.
3. AI에게 지시만으로 **실행 가능한 풀스택 앱**(FastAPI + Streamlit + PostgreSQL)을 만든다.
4. 만든 앱을 **공개 도메인으로 배포**하여 다른 사람이 화면으로 체험하게 한다.

---

## 1. 바이브코딩이란?

**바이브코딩** = 코드를 한 줄씩 직접 타이핑하는 대신, **자연어 프롬프트로 의도를 전달**하고
AI가 코드를 생성·수정하게 하면서, **실행 결과를 보고 다시 지시**하는 개발 방식입니다.

핵심은 두 가지입니다.

- **프롬프트 = 명세서(Spec)**: 무엇을, 어떤 제약으로, 어떤 결과물을 원하는지 글로 정의한다.
- **반복 루프(Loop)**: `프롬프트 → 생성 → 실행 → 오류/부족 발견 → 다시 프롬프트` 를 반복한다.

> 💡 코드를 모르더라도, **"무엇이 필요한지 정확히 말할 수 있으면"** 만들 수 있습니다.
> 그래서 바이브코딩의 실력 = **프롬프트(요구사항)를 잘 쓰는 능력**입니다.

## 2. ⭐ 바이브코딩 프롬프트의 필요충분조건

이 실습의 가장 중요한 학습 포인트입니다. 아래 12개 항목을 체크리스트로 사용하세요.

### 2-1. 필요조건 (이것이 없으면 결과가 망가진다)

| # | 조건 | 이유 / 나쁜 예 → 좋은 예 |
|---|------|--------------------------|
| 1 | **목표·산출물 명확화** | "주식 앱 만들어줘"(X) → "종목코드 입력 시 분석 점수·PDF·화면을 내는 MVP"(O) |
| 2 | **기술 스택·제약 명시** | "Python/FastAPI/Streamlit/PostgreSQL 사용" 처럼 환경을 못 박는다 |
| 3 | **폴더구조·파일목록 지정** | 재현성을 위해 만들 파일을 나열한다(AI가 임의로 흩어놓지 않음) |
| 4 | **보안 원칙** | "API 키는 .env에서만 읽고, 화면/저장소에 노출 금지" |
| 5 | **오류 내성** | "키가 없어도 앱이 죽지 않고 친절한 메시지" — 실패해도 멈추지 않게 |
| 6 | **실행 가능성** | "비개발자가 그대로 실행 가능하게, Windows 실행순서 README 포함" |

### 2-2. 충분조건 (이것이 있으면 결과가 완성도 높다)

| # | 조건 | 이유 |
|---|------|------|
| 7 | **검증 가능한 수용 기준** | "GET /report/005930 이 점수+브리프를 반환" 처럼 합격 조건을 준다 |
| 8 | **샘플/목업 fallback** | 외부 API 키 없이도 동작(샘플 데이터, mock 브리프) → 항상 데모 가능 |
| 9 | **문서화 요구** | README·주석·고지문 등 산출물에 설명을 포함시킨다 |
| 10 | **테스트 포함** | "외부 호출 없는 단위 테스트 작성" → 회귀 방지 |
| 11 | **배포 경로 제공** | Dockerfile·배포 설정 → 실제 서비스로 연결 |
| 12 | **피드백 루프 설계** | "실행하라", "화면으로 보여달라" 처럼 결과를 보고 다음을 지시 |

> ✅ **필요조건**은 "동작하는 결과"를 보장하고,
> **충분조건**은 "신뢰할 수 있고 배포 가능한 결과"를 보장합니다.

### 2-3. 좋은 프롬프트 한 줄 공식

```
[목표/산출물] + [스택/제약] + [폴더·파일 구조] + [보안·오류원칙]
+ [샘플 fallback] + [검증기준] + [문서·테스트·배포] 요구
```

## 4. Step 1 — 최초 프롬프트로 프로젝트 생성

### 무엇을 했나
단 하나의 큰 프롬프트로 약 48개 파일(수집기·분석·스코어링·GPT 브리프·PDF·FastAPI·Streamlit·
텔레그램·스케줄러·테스트·Docker·README)을 한 번에 생성했습니다.

### 왜 잘 됐나 (2장 조건 적용 확인)
- 목표/산출물 명확(점수+브리프+PDF+화면), 스택 명시(FastAPI/Streamlit/PG)
- 폴더구조를 직접 나열 → 재현성 확보
- "키 없어도 죽지 않게", ".env만 사용", "샘플 데이터로 동작" → 오류 내성 + fallback
- "README에 Windows 실행순서", "테스트 포함" → 문서·테스트

### 교육생 실습
프로젝트 루트 README 상단의 요구사항 블록(또는 강사 제공 프롬프트)을 그대로 AI에게 제시하세요.
생성이 끝나면 다음을 확인합니다.
- [ ] `KOStockCrewAI/` 폴더와 하위 파일이 생성되었는가
- [ ] `.env.example` 은 있고 `.env` 는 없는가(보안)
- [ ] `requirements.txt`, `README.md`, `Dockerfile` 이 있는가

## 5. Step 2 — 실행하고 검증하기

### 프롬프트 예
> "실행하라" / "샘플 데이터로 /report/005930 을 테스트하라"

### 따라하기 (로컬)
```bash
python -m venv venv && venv\Scripts\activate      # (Windows)
pip install -r requirements.txt
docker compose up -d postgres                       # DB 실행
uvicorn main:app --reload                           # API 실행
```
브라우저에서:
- `http://127.0.0.1:8000/docs` → **GET /sample/load/005930** 실행
- **GET /report/005930** → 점수 + 브리프 JSON 확인

### 핵심 교훈
**"실행하라"는 가장 강력한 프롬프트 중 하나**입니다. 실행하면 오류가 드러나고,
오류 메시지를 그대로 AI에게 주면 수정으로 이어집니다(반복 루프).

## 6. Step 3 — 기능 보강 (스케줄러 실제 구현)

### 프롬프트 예
> "스케줄러의 placeholder(TODO)를 실제 collector→repository 연결 로직으로 채워라.

처음엔 **뼈대(placeholder)**만 만들고, 이후 프롬프트로 **살을 붙입니다**.
"한 번에 완벽" 대신 "동작하는 최소 → 점진적 강화"가 바이브코딩의 정석입니다.

## 7. Step 4 — 공개 배포 준비

### 프롬프트 예
> "공개 접속이 가능하도록 배포 설정을 생성하라. 기존 앱은 건드리지 말라."

### 결과물
- `render.yaml` (Render: API+UI+PostgreSQL 한 번에)
- `railway.json` (Railway)
- `fly.toml` (Fly.io)
- 앱이 시작 시 DB 테이블 자동 생성(`init_db`), 클라우드 DB 주소 자동 정규화

### 교훈
배포는 "코드 완성 후 별도 작업"이 아니라 **프롬프트로 함께 요청**할 수 있습니다.
단, 배포 플랫폼별 제약(서버리스 vs 컨테이너)을 AI가 설명하게 하여 올바른 선택을 합니다.

## 8. Step 5 — 진단 도구·테스트 추가

**테스트와 진단은 "신뢰"를 만듭니다.** 충분조건(2장 #7,#10)을 충족시키는 단계입니다.

## 10. Step 7 — 사용자 UI 다단계화

### 프롬프트 예
> "① 정상 운용 초화면 ② API 키 설정(등록/수정/완료여부) ③ 종목명·코드 입력 1페이지 리포트
>  ④ PDF 다운로드 메뉴 — 4가지를 갖춘 사용자 화면으로 만들라. 키 값은 화면에 표시하지 말라."

### 결과물 (Streamlit 3개 메뉴)
- 🏠 **홈**: 서버/DB/키 상태 카드
- 🔑 **API 키 설정**: 마스킹 입력 등록/수정, ✅설정됨/⛔미설정, 실연결 점검
- 📈 **리포트 조회**: 종목명("삼성전자")→코드("005930") 자동 변환 → 1페이지 리포트 → PDF 다운로드

### 교훈
요구사항을 **번호로 나눠 명확히** 주면(2장 #1,#7) AI가 빠짐없이 구현합니다.
보안 제약("키 표시 금지")을 함께 주는 것이 필요조건(2장 #4)입니다.

## 12. 🌐 배포해서 체험하기 (교육생용 도메인 발급)

> 목표: 강사 또는 각 교육생이 **공개 https 도메인**을 발급받아, 누구나 브라우저로 UI를 체험.

### 가장 쉬운 방법 — Render Blueprint (API+UI+DB 한 번에)
1. https://render.com 가입 → GitHub 연결
2. **New + → Blueprint** → 이 저장소(`inan1105/HQ_Chart`)와 브랜치 선택
3. **Apply** → 잠시 후 두 개의 공개 URL 발급
   - UI(화면): `https://kostock-ui-XXXX.onrender.com`
   - API/문서: `https://kostock-api-XXXX.onrender.com/docs`
4. (선택) `kostock-api` 서비스 **Environment** 에 실제 API 키 입력 → 재시작
   - 키가 없어도 **샘플 모드**로 모든 화면 체험 가능

> 한 줄 배포 링크(기본 브랜치 기준):
> `https://render.com/deploy?repo=https://github.com/inan1105/HQ_Chart`

### 다른 방법
- **Railway**: 루트 디렉토리를 `KOStockCrewAI` 로, Postgres 추가, 도메인 생성 (`railway.json` 사용)
- **Fly.io**: `KOStockCrewAI` 폴더에서 `fly launch` → `fly postgres create/attach` → `fly deploy` (`fly.toml`)

자세한 절차는 `README.md` 의 "공개 인터넷 배포" 및 "다른 호스트로 배포" 절을 참고하세요.

### 교육 운영 팁
- **강사 1명이 1개 인스턴스 배포** → URL을 교육생 전원에게 공유 → 모두 같은 화면 체험
- **심화**: 교육생이 각자 자신의 GitHub 포크에 배포해 보며 "내 손으로 배포" 경험

> ⚠️ 안내: 본 앱의 결과는 **정보 제공용이며 투자 권유가 아닙니다.**
> 샘플 데이터는 실데이터가 아닌 검증용입니다. DART/ECOS/코스콤 이용약관을 준수하세요.
